# Сделай чек-лист

Ниже — практический **чек-лист приемки реализации по этапам P0–P9** для вашего проекта. Он заточен не под “код вообще работает”, а под проверку, что Cursor реализует именно **вашу архитектуру MVP**, а не просто пишет набор файлов.

***

# Чек-лист приемки реализации MVP

## Как использовать

Для каждого этапа проверяйте 4 слоя:

- **Scope** — сделано ли именно то, что было запланировано.
- **Architecture** — не нарушены ли ключевые границы.
- **Behavior** — работает ли система в пользовательском смысле.
- **Demo readiness** — помогает ли это будущему показу, а не мешает ему.

Рекомендуемый статус по этапу:

- **Accepted** — можно идти дальше.
- **Accepted with notes** — идти можно, но есть техдолг.
- **Rejected** — этап нужно доработать до перехода дальше.

# P0 — Bootstrap

## Scope

- Есть `docker-compose.yml`.
- Есть `.env.example`.
- Есть `.gitignore`.
- Созданы каталоги `backend/`, `ui/`, `ha/`, `tools/`.
- Есть минимальный `README.md` с запуском.
- Есть минимальный backend app с health endpoint.
- Есть базовый UI skeleton.
- Есть HA configuration skeleton.


## Architecture

- Структура проекта соответствует плану, а не импровизирована на лету.
- Пути и каталоги не конфликтуют с docs и plan.
- Нет лишних технологий, которых не было в плане.
- Нет premature optimization: никаких RabbitMQ, Redis, Celery, Postgres “на будущее”, если они не нужны сейчас.


## Behavior

- Проект поднимается локально одной командой.
- Backend отвечает health-check.
- UI открывается.
- HA контейнер стартует без явных критических ошибок конфигурации.


## Demo readiness

- Новый разработчик может запустить каркас по README без устных пояснений.
- Нет ручной магии вроде “сначала зайди в контейнер и там что-то допиши”.
- Порядок старта сервисов устойчивый.


## Red flags

- В `docker-compose` слишком много сервисов.
- README не соответствует реальному запуску.
- Backend или UI живут вне общей локальной сборки.
- HA не стартует стабильно.


## Вердикт этапа

Этап принимается только если каркас реально поднимается и не создает хаос для следующих фаз.

# P1 — Backend API baseline

## Scope

- Объявлены основные API endpoints.
- Есть Pydantic/typed models.
- Есть in-memory event log skeleton.
- Есть scenario engine skeleton.
- Есть baseline tests.


## Architecture

- Route handlers тонкие.
- Бизнес-логика не в endpoints.
- Event log вынесен в service.
- Scenario engine пока skeleton, но не превращен в god object.
- Контракты моделей читаемые и пригодны для роста.


## Behavior

- Health endpoint работает.
- `GET state` возвращает валидный stub/empty response.
- `GET events` возвращает пустой или базовый список.
- `POST execute` возвращает контролируемый “not implemented” или корректный stub.
- Ошибки API оформлены консистентно.


## Demo readiness

- Уже можно подключать UI без хаотических заглушек.
- Структура ответов не выглядит временной и одноразовой.


## Red flags

- Логика уже поползла в routes.
- Нет нормальной типизации ответов.
- Error responses разнородны.
- Контракты названы слишком абстрактно или нестабильно.


## Вердикт этапа

Этап принимается, если API skeleton уже можно считать фундаментом, а не временной импровизацией.

# P2 — Home Assistant virtual house

## Scope

- Есть 3 комнаты.
- Есть набор виртуальных устройств по комнатам.
- Есть сцены.
- Есть custom sentences.
- Есть intent/conversation wiring, если планом предусмотрено.


## Architecture

- HA действительно используется как source of truth.
- YAML не содержит бизнес-логики, которая должна жить в backend.
- Сущности названы системно и предсказуемо.
- Комнатная модель читается по entity naming.
- Нет дублирующих сущностей с неясной ролью.


## Behavior

- HA показывает сущности и их состояния.
- Сцены реально меняют состояние устройств.
- Базовые устройства доступны через HA API.
- Конфигурация не разваливается на reload/restart.


## Demo readiness

- Дом выглядит убедительно, а не как случайный набор demo entities.
- Есть минимум один сценарий, который меняет несколько состояний сразу.
- Сущности удобно объяснять на защите.


## Red flags

- Слишком много сущностей “на будущее”.
- Несистемные entity IDs.
- Логика комнат и устройств размыта.
- YAML начинает заменять backend logic.


## Вердикт этапа

Этап принимается, если виртуальный дом уже можно использовать как реальный baseline для backend и демо.

# P3 — Backend ↔ HA integration

## Scope

- Backend умеет читать состояние дома из HA.
- Backend умеет писать в HA для простых действий.
- `GET /api/state/house` строится из HA.
- Нет локального дубля состояния в backend.


## Architecture

- HA client выделен отдельно.
- Нормализация house state вынесена в отдельный слой.
- Backend не хранит вторую “истину”.
- Ошибки интеграции с HA обрабатываются явно.


## Behavior

- После изменения в HA backend отдает обновленное состояние.
- Простые команды через backend реально меняют HA state.
- При недоступности HA backend дает контролируемую ошибку.
- Таймауты и connection errors не приводят к падению всего приложения.


## Demo readiness

- Можно показать, что UI работает не на моках, а на реальном состоянии дома.
- Изменение состояния отражается сквозным образом: backend → HA → state response.


## Red flags

- Backend кэширует состояние как основной источник.
- Есть параллельная in-memory модель дома.
- Маппинг entity → domain model хаотичный.
- Ошибки HA “проглатываются”.


## Вердикт этапа

Этап принимается только если HA действительно становится operational source of truth.

# P4a — Execution core

## Scope

- Реализованы `turn_on_device`, `turn_off_device`, `activate_scene`.
- Есть единый `ExecutionResponse`.
- Есть `DebugTrace`.
- Есть `AffectedEntity`, `StateChange`, `ScenarioError`.
- Есть `response_builder`.
- Есть минимальный `reset`.
- Есть execution tests.


## Architecture

- Есть intent registry/router.
- Есть handlers.
- Route handlers не содержат business logic.
- `scenario_engine` orchestrates, а не делает всё сам.
- Есть alias/entity resolver до исполнения.
- Event log интегрирован централизованно.


## Behavior

- Команда на включение реально включает устройство.
- Команда на выключение реально выключает устройство.
- Сцена реально меняет несколько состояний.
- Ответы консистентны по форме.
- `DebugTrace` полезен, а не формальный.
- `reset` возвращает систему в baseline.
- После команды `GET /api/state/house` показывает новое состояние.


## Demo readiness

- Уже можно провести первый текстовый demo flow.
- Видно, что команда дошла до устройства и вызвала состояние.
- Reset позволяет быстро повторять сценарии.


## Red flags

- Большой `if/elif` вместо реестра handler’ов.
- Route handlers толстые.
- Resolver отсутствует или смешан с ambiguity.
- Ответы у разных интентов разной формы.
- Reset сделан костылем и не всегда возвращает baseline.


## Acceptance

- Можно выполнить минимум 3–5 базовых команд подряд.
- После reset система воспроизводимо возвращается в исходное состояние.

# P4b — Status layer

## Scope

- Реализованы `get_room_status`, `get_device_status`, `get_sensor_status` хотя бы в минимальном объеме.
- Есть tests на status queries.
- Status responses проходят через общий response family.


## Architecture

- Status layer отделен от execution core.
- Read-only запросы не используют side effects.
- Агрегация статусов вынесена в отдельный service.
- Formatting status response не смешан с imperative logic.


## Behavior

- Запрос статуса комнаты возвращает корректный агрегированный ответ.
- Запрос статуса устройства возвращает корректный ответ.
- Запрос датчика возвращает корректный ответ.
- Повторный одинаковый запрос при неизменном state дает согласованный результат.


## Demo readiness

- Статусные вопросы выглядят естественно и полезно.
- Ответы короткие и понятные.
- Можно показать “дом не только управляется, но и репортит состояние”.


## Red flags

- Status query случайно меняет состояние.
- Ответы слишком многословные.
- Status logic продублирована в UI.
- Status layer снова запихнут в scenario engine без границ.


## Вердикт этапа

Этап принимается, если read-path зрелый и не ломает execution architecture.

***

# P5 — Ambiguity, clarify, compound

## Scope

- Есть ambiguity resolver.
- Есть `clarify` endpoint/flow.
- Есть continuation model.
- Есть session handling.
- Есть TTL.
- Есть compound action support.
- Есть тесты на ambiguity/clarify/TTL.


## Architecture

- Ambiguity работает по canonical candidates, а не по сырым строкам.
- Continuation context хранится отдельно и предсказуемо.
- Clarification flow не размазан по UI и backend хаотично.
- Compound actions не ломают единый response contract.


## Behavior

- Неоднозначная команда не исполняется молча.
- Система просит уточнение.
- Уточнение корректно продолжает предыдущий intent.
- Просроченный clarify context корректно отклоняется.
- Неверный clarify payload обрабатывается безопасно.
- Compound action либо выполняется контролируемо, либо дает понятную ошибку.


## Demo readiness

- Есть как минимум один сильный ambiguity case для показа.
- Уточнение проходит естественно.
- Лог событий и debug trace помогают объяснить поведение системы.


## Red flags

- Clarify flow работает только “вручную” через UI-хак.
- Session handling нестабильно.
- Ambiguity resolver завязан на raw strings.
- Compound action сделан через ad hoc if/else без модели.


## Acceptance

- Команда типа “Выключи свет” приводит к уточнению.
- После уточнения выполняется правильное действие.
- Истекший session/TTL обрабатывается явно.
- Есть минимум один compound scenario.

# P6a — Functional UI

## Scope

- Есть dashboard.
- Есть комнаты/устройства.
- Есть event log.
- Есть command console.


## Architecture

- UI берет состояние только с backend.
- UI не становится вторым source of truth.
- Компоненты разделены осмысленно.
- API client централизован.


## Behavior

- UI показывает текущее состояние дома.
- После команды состояние обновляется.
- Event log отображается.
- Команды из консоли доходят до backend.
- UI не ломается при error response.


## Demo readiness

- Уже можно показать управление домом без operator tooling.
- Интерфейс читаемый и не стыдный для демо.
- Видна связь “команда → изменение состояния”.


## Red flags

- UI напрямую ходит в HA в обход backend.
- Состояние локально “дорисовывается” без подтверждения backend.
- Нет явных состояний loading/error/empty.
- UI слишком сырой для восприятия.


## Вердикт этапа

Этап принимается, если есть рабочий presentation layer, а не просто набор React-компонентов.

# P6b — Operator / demo UI

## Scope

- Есть clarification panel.
- Есть debug trace panel.
- Есть demo controls.
- Есть failure visibility.


## Architecture

- Operator UI не ломает основной UI.
- Debug trace отображает данные из backend contract, а не синтетическую фронтовую интерпретацию.
- Demo controls используют backend APIs, а не фронтовые костыли.


## Behavior

- При clarification response UI показывает панель уточнения.
- Debug trace соответствует последней команде.
- Reset доступен из UI.
- Ошибки и сбои видны оператору.


## Demo readiness

- Оператор может быстро восстановить baseline.
- Во время демонстрации можно объяснить, что произошло.
- UI помогает при сбое, а не мешает.


## Red flags

- Clarification panel не синхронизирована с session flow.
- Debug trace не соответствует реальному backend response.
- Demo controls работают нестабильно.
- Operator tooling встроен хаотично.


## Вердикт этапа

Этап принимается, если UI реально поддерживает демо и отладку, а не только красиво выглядит.

# P7 — Demo hardening

## Scope

- Есть replay.
- Есть set-mode.
- Есть deterministic demo mode.
- Reset расширен до полноценного demo-control behavior.


## Architecture

- Demo controller выделен отдельно.
- Demo mode не ломает основную логику MVP.
- Replay не зашит “одноразовым скриптом в UI”.
- Режимы сформулированы ясно и проверяемо.


## Behavior

- Reset всегда возвращает baseline.
- Replay воспроизводим.
- Set-mode реально меняет режим работы так, как описано.
- Event log и UI синхронизированы после reset/replay.


## Demo readiness

- Команда может за 10–20 секунд вернуть систему в демонстрационное состояние.
- Повторный прогон сценариев дает тот же результат.
- Сбой не разрушает весь показ.


## Red flags

- Replay работает только в одном счастливом кейсе.
- Reset иногда оставляет “грязное” состояние.
- Set-mode не влияет ни на что реальное.
- Demo behavior живет отдельно от основной архитектуры.


## Вердикт этапа

Этап принимается, если проект становится устойчивым к повторяемой демонстрации.

# P8 — Tests and docs sync

## Scope

- Есть поведенческие тесты на критические сценарии.
- Обновлены docs.
- README соответствует реальности.
- API contracts синхронизированы с кодом.
- Demo script соответствует поведению системы.


## Architecture

- Тесты проверяют behavior, а не только внутренние функции.
- Документация не противоречит реализации.
- Acceptance criteria соотносятся с тем, что реально сделано.


## Behavior

- Основные сценарии проходят тестами.
- Docs помогают реально поднять и понять проект.
- Demo script можно исполнить по шагам.


## Demo readiness

- Есть понятный сценарий показа.
- Есть указание fallback-пути.
- Есть documented recovery path.


## Red flags

- Docs обновлены формально.
- Тесты покрывают только happy path.
- README устарел.
- Demo script не соответствует актуальному UI/API.


## Вердикт этапа

Этап принимается, если проект можно передать третьему человеку и он сможет понять, поднять и показать систему.

# P9 — Voice optional

## Scope

- Голос добавлен как опциональное расширение.
- Голос не ломает text-first путь.
- Есть fallback на текст.


## Architecture

- Voice path подключен поверх существующей системы.
- Нет переписывания core architecture ради голоса.
- Голос не превращен в обязательную зависимость MVP.


## Behavior

- Минимальный voice flow работает или безопасно подготовлен.
- При проблеме голоса текстовый сценарий остается рабочим.
- Voice responses не ломают существующий response contract.


## Demo readiness

- Голос можно показать, если стабилен.
- Если нестабилен, его можно отключить без разрушения демо.
- Команда знает, когда переходить на text fallback.


## Red flags

- Ради голоса изменен backend contract.
- Голос ломает demo stability.
- Wake-word/STT/TTS начинают доминировать над MVP.
- Текстовый путь деградировал.


## Вердикт этапа

Этап принимается только как безопасное расширение, но не как условие успеха MVP.

# Сквозной чек-лист по всей реализации

## Архитектурные инварианты

- HA остается единственным source of truth.
- Backend владеет business logic.
- UI не хранит отдельную истину.
- YAML остается integration/config layer.
- Text-first сохраняется до конца.
- Voice не ломает MVP.


## Качество реализации

- Нет толстых route handlers.
- Нет god object в `scenario_engine`.
- Есть централизованный response contract.
- Есть event log.
- Есть reset.
- Есть clear error handling.
- Есть deterministic demo path.


## Продуктовая зрелость

- Есть базовое управление.
- Есть статусные запросы.
- Есть сцены.
- Есть ambiguity handling.
- Есть fallback.
- Есть operator visibility.
- Есть demo recoverability.


## Что считается “не принимать”

- Дублирование state между HA и backend.
- Размазанная логика по routes/UI/YAML.
- Отсутствие reset.
- Нет clarify flow при неоднозначности.
- Нет event log/debug trace.
- Голос ломает text-first MVP.
- Демо зависит от ручной подправки состояния.

# Финальный чек-лист перед демонстрацией

## Обязательно

- Compose поднимается с нуля.
- HA стабильно стартует.
- Backend видит HA.
- UI видит backend.
- Работает on/off.
- Работает хотя бы одна сцена.
- Работает хотя бы один status query.
- Работает ambiguity case.
- Работает reset.
- Event log отражает действия.
- Есть text fallback.


## Желательно

- Работает compound action.
- Есть debug trace panel.
- Есть replay.
- Есть demo mode indicator.
- Есть documented demo script.


## Нельзя выходить на показ, если

- reset нестабилен;
- UI иногда показывает не то состояние;
- clarify flow ломается;
- state расходится между HA и backend;
- README/demo script не соответствуют системе;
- без ручных костылей нельзя повторить сценарий.
